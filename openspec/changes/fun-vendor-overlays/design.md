## Context

`fun build` today hardcodes vendor file lists in `arch/build/<platform>/generator.c`. This worked while the only vendor was `fundamental`, but breaks whenever modules are added (the infinite-loop bug that triggered this change), and has no story for third-party libraries dropped into `vendor/`. The tokenizer (Phase 1a of `fun-c-compiler`) is complete and available for include-directive scanning.

## Goals / Non-Goals

**Goals:**
- `fun build` resolves vendor file lists automatically for any library in `vendor/`
- `vendor/.fun/<libname>.toml` is the single source of truth for how to build a vendored library
- Overlays for fundamental are auto-generated and cached; no manual maintenance required
- Community overlays for popular libraries can be fetched from a separate registry
- Symbol-resolution fallback is designed for but gated on compiler Phase 2

**Non-Goals:**
- Building an overlay authoring UI or CLI wizard
- Hosting or operating the overlay registry (protocol only)
- Full compiler integration (symbol-resolution fallback is future work)
- Windows-only: all paths and platform keys must support linux-amd64 too

## Decisions

### D1: Overlay location — `vendor/.fun/<libname>.toml`

Overlays live inside `vendor/` but in a hidden subdirectory, keeping them co-located with the libraries they describe without polluting library directories.

```
vendor/
  fundamental/       ← library, untouched
  libpng/            ← library, untouched
  .fun/
    fundamental.toml ← auto-generated by fun build
    libpng.toml      ← fetched from registry or hand-authored
```

**Why not inside the library dir** (`vendor/libpng/fun.toml`)? Keeps upstream source trees pristine — no foreign files mixed with library source. Matches the DefinitelyTyped model where `@types/foo` never modifies `node_modules/foo`.

**Why not `.fun/vendor/`** at project root? Keeps everything vendor-related inside `vendor/`. A project-root `.fun/` directory will eventually hold cache and compiler artifacts — mixing overlay metadata there conflates concerns.

### D2: Three-tier resolution, in order

```
Tier 1 — Overlay present
  Read vendor/.fun/<libname>.toml
  Use sources/includes/platform sections directly
  Fast: no file scanning

Tier 2 — No overlay, fundamental-style library
  Detect: library has vendor/<lib>/include/<lib>/...  structure
  Scan all src/*.c for #include "lib/..." directives (tokenizer)
  Walk vendor/<lib>/include/<lib>/.../*.h transitively for sub-deps
  Collect module names → expand to src/<mod>/*.c + arch/<mod>/<plat>/*.c
  Write vendor/.fun/<libname>.toml (cache for next build)

Tier 3 — No overlay, unknown library (future, gated)
  Requires fun-c-compiler Phase 2 (parser + top-level decl extraction)
  Parse src/*.c → collect undefined symbol references
  Scan vendor/<lib>/**/*.c for definitions
  Resolve transitively until fixed point
  Write vendor/.fun/<libname>.toml
  Gate: if compiler not available, warn and skip library
```

**Why write the overlay after Tier 2/3?** Subsequent builds skip scanning entirely. The generated overlay is commitable — teammates don't re-scan. It also makes the resolution auditable: you can read `vendor/.fun/fundamental.toml` to see exactly what was determined.

### D3: Overlay format — TOML subset

```toml
# vendor/.fun/fundamental.toml
[library]
name = "fundamental"
generated = true          # auto-generated, may be overwritten

[build]
sources = [
  "src/async/async.c",
  "src/config/config.c",
  # ...
]
includes = ["include"]

[build.windows-amd64]
sources_extra = [
  "arch/async/windows-amd64/async.c",
  # ...
]

[build.linux-amd64]
sources_extra = [
  "arch/async/linux-amd64/async.c",
  # ...
]

[dependencies]
# sub-dependencies that must also be resolved
# (empty for fundamental; libpng might list zlib here)
```

All paths in the overlay are relative to the library root (`vendor/<libname>/`).

**Why TOML?** The existing `fun.ini` is INI format. TOML extends INI cleanly with arrays and sections — the parser can share the ini parser infrastructure. No new parser dependency.

### D4: Startup is always-inject, not overlay-resolved

`vendor/fundamental/src/startup/startup.c` and `vendor/fundamental/arch/startup/<platform>/windows.c` have no corresponding `#include` directive anywhere. They define the entry point. Always inject these unconditionally before the resolved file list.

### D5: Overlay registry — separate HTTPS endpoint, read-only

```
https://overlays.fun-pkg.dev/<libname>/<version>/fun.vendor.toml
```

`fun add <lib>` (from fun-package-manager) attempts a GET after vendoring. 404 = no overlay, continue silently. No auth. No write access from the client. Registry is operated separately; this change only specifies the fetch protocol.

**Why separate from package registry?** Overlays and packages have different lifecycles. A libpng overlay for 1.6.40 might be updated independently of the library. Separate registries allow separate maintainers, separate versioning, separate governance — exactly the DefinitelyTyped model.

### D6: `arch/build/<platform>/generator.c` calls resolver, not hardcoded list

The generator's hardcoded vendor string is replaced with a call to `vendor_resolve_all()` which returns a space-separated file list. The generator assembles the gcc command as before, just with dynamic content.

## Risks / Trade-offs

**[Risk] Generated overlay goes stale when fundamental gains new modules** → Mitigation: `fun build` checks mtime of `vendor/.fun/fundamental.toml` against `vendor/fundamental/` directory mtime; regenerates if stale. User can also delete the overlay to force regeneration.

**[Risk] Tier 2 include-scan misses transitive vendor-internal dependencies** (e.g., `config.c` includes `hashmap.h` internally, not from user code) → Mitigation: scan vendor headers transitively, not just user source. The BFS over vendor include headers catches this.

**[Risk] TOML parser adds scope** → Mitigation: the overlay format is a strict subset — no inline tables, no arrays of tables. The existing ini parser handles `key = "value"` and `[section]`; extend it to handle `key = ["a", "b"]` array syntax only.

**[Risk] Tier 3 (symbol resolution) raises expectations before compiler is ready** → Mitigation: Tier 3 is explicitly gated. `fun build` warns "no overlay found for <lib>, skipping" with a hint to create one. No silent failure.

## Open Questions

- **Overlay registry domain**: `overlays.fun-pkg.dev` is a placeholder. Actual hosting TBD.
- **Overlay versioning**: Should the fetch URL include the library version? Overlays may need to change when library internals change (new `.c` files). Lean yes.
- **Stale detection granularity**: Directory mtime is coarse. Content hash of the vendor dir would be more correct but expensive. Start with mtime, revisit.
